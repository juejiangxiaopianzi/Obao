# Obao · 飞书周报审阅闭环（副本承载页）

解决一个根本问题：**静态本地 HTML 你改了，agent 看不见、没有回路。** 飞书是双向平台——agent 能写、你能改、agent 还能读回你改了啥。所以用**飞书文档副本**当承载页，你的「删 / 改 / 回复」就是纠正信号，agent 读得回来。

## 读配置 + 前置（必做）

- 读 `~/.obao-review/corrections.md`（历史纠错 · 不存在就跳过）+ 用户的分身设定。
- **lark-cli 前置**：`which lark-cli`，没装 Obao 自己 `npm i -g @larksuite/cli`；未登录提示 `!lark-cli auth login`。评论/复制需要 scope，缺了按报错提示 `lark-cli auth login --scope "..."` 引导授权一次。

## 触发

「帮我审这份飞书周报 [链接]」「飞书审周报」「副本审阅」——进 Stage A。
「可以推送了」「推吧」——进 Stage B（针对最近一个未推送的副本 loop）。

---

## Stage A · 审 + 建副本 + 草评（把追问标到副本上）

### A1 · 拿齐输入
- **周报**：用户给的飞书文档 URL → `lark-cli docs +fetch --as user --doc <url>` 拿正文。
- **OGSM / OKR**：用户这次给的，或已存 `~/.obao-review/ogsm.md`（一次性，多份周报可共用一份）。没有就问，不许编。
- **分身设定**：用户的视角/红线（一段话）。

### A2 · 审（用 obao-review 的方法论 + 主动套用案例库）
**开局先读 `~/.obao-review/corrections.md`（案例库 bad→good）并主动套用**：审每条之前，扫一遍案例——
- 命中某条的「触发场景」→ **直接用它的 ✅ good 问法**，别再问 ❌ bad 那种。
- 案例库记的是「使用者真正关心什么」（如：试点/预存类→问能不能规模化复制，不是问客户说了啥）。这是越用越准的来源，必须用，不是摆设。

然后按 D1-D7 + 化词评论 + 自动 @ 识别，把周报审成 N 条追问，每条 =
`{ anchor: 原文里 20-60 字的连续独特片段, owner: @谁, text: 化词评论草稿 }`。
- `anchor` 必须是原文真实存在的连续片段（含独特数字/专名），后面划词定位靠它。
- `text` 守化词评论：口语、引原文/KR 原值、3-5 个**直击核心**的追问（套案例库的 good 问法）、不替下属拟方案、不借上级虎皮。
- ⛔ **评论里只放问题本身**，绝不写「我之前问偏了 / 这不是重点 / 换个问法」这类 Obao 自述——下属看的是干净的追问。

### A3 · 建副本（fetch → create）
- 用 A1 fetch 到的正文，`lark-cli docs +create --as user --title "<原标题> · Obao审阅副本【只动评论别动正文】" --markdown @-`（stdin 喂正文）。
- 拿到副本 `doc_url` / `doc_token`。

### A4 · 在副本上逐条划词评论（草稿态 · @ 不通知真人 · 幂等防重）
- ⛔ **幂等防重（必做）**：发之前先 `lark-cli drive file.comments list`（params 传 file_token+file_type）拿副本现有评论；**同一个 anchor 已经有 Obao 评论就跳过**，不重复发。批量发**逐条串行 + 每条间隔约 1s**，别并发（并发会丢、会重）。
- 逐条 `add-comment`：
```bash
lark-cli drive +add-comment --as user --doc <副本token> --type docx \
  --selection-with-ellipsis "<anchor>" \
  --content '[{"type":"text","text":"@<owner> <text>"}]'
```
- ⚠️ `--type docx` 必带（裸 token 必须给 type）。
- ⚠️ **副本里的 @ 用纯文字**（`{"type":"text"}` 里写「@林深」），**不要用真 mention 元素** → 不通知本人，纯草稿态。
- 记下每条返回的 `comment_id`（解析整段 JSON，别只取最后一行）。

### A5 · 存底稿（Stage B 要拿它做对照）
把这次 loop 写进 `~/.obao-review/loops/<副本token>.json`：
```json
{
  "copy_token": "<副本token>",
  "copy_url": "...",
  "origin_token": "<原周报token>",
  "origin_url": "...",
  "created_at": "...",
  "status": "pending_review",
  "comments": [
    { "comment_id": "...", "anchor": "...", "owner": "林深", "original_text": "@林深 ..." }
  ]
}
```

### A6 · 把球交给用户
给副本链接 + 说明审法：
> "我在**副本**上标了 N 条追问（@ 只写了名字、没通知人）：{副本链接}
> 你过一遍——**不动的 = 我直接发；删掉的 = 不发；改了或回复了 = 你在纠正我**。
> 弄完跟我说一句『可以推送了』，我就读回你的改动、把该发的发到**原周报**（这次真 @ 通知）。"

## Stage B · 「可以推送了」→ 读回轨迹 → 发原周报

### B1 · 读回副本**全部**评论（按锚句匹配，不只按底稿 id）
```bash
lark-cli drive file.comments list --as user \
  --params '{"file_token":"<副本token>","file_type":"docx"}' --page-all
```
- 解析：`data.items[]`，每条有 `comment_id` / `quote`（锚句）/ `is_solved` / `reply_list.replies[]`（第 0 条=Obao 原评论，之后=用户回复）。
- ⛔ **必须读全部评论、按 `quote`（锚句）匹配回底稿的事项**，不要只遍历底稿的 comment_id。（否则用户回复落在某条重复评论/非底稿条上就漏看——这是真踩过的坑。）

### B2 · 逐条判定（核心）
对每个底稿事项，找副本上**锚到同一句**的评论，看它现状：

| 副本里的现状 | 判定 | 动作 |
|---|---|---|
| 还在 · 没改 · 无用户回复 | **认可** | 原样发到原周报 |
| 不在了（被删） | **否决** | **跳过，绝不发** |
| Obao 原评论被编辑 | **采纳改动** | 用改后的内容发到原周报 |
| 有用户回复（replies 里多了非 Obao 的） | **纠正** | 见下方「纠正处理」 |

**纠正处理（重点）**：
1. **先消化用户回复的真实意图**——他在说「我真正想问的是 X，你问偏了」。提炼出他关心的核心。
2. **重写这条评论 = 只输出那个 good 问法本身**。⛔ 绝不写「我之前问偏了 / 这不是重点 / 换个问法」之类自述——下属看的是干净的追问。
3. 发重写版到原周报；如果原来已经把 bad 版发过去了，**把 bad 版 `patch` 成 is_solved 隐去**，不留两版。
4. **沉淀成结构化案例**（见 B4），不是一句流水账。

> 既改又回复 → 以回复的意图为准重写。

### B3 · 发到原周报（这次真 @ 通知）
对每条「要发」的，在**原周报**（不是副本）上 add-comment：
```bash
lark-cli drive +add-comment --as user --doc <原周报token> \
  --selection-with-ellipsis "<anchor>" \
  --content '[{"type":"text","text":"<最终评论文>"}]'
```
- **真 @ 通知**：尽量把 owner 解析成飞书用户再用 mention 元素（`lark-cli contact +search` 拿 open_id → `{"type":"mention","user_id":"ou_..."}`）；解析不到就退回纯文字「@名字」。
- anchor 还用底稿里那条（定位到原文同一句）。

### B4 · 学习（被纠正的沉淀成**结构化案例** · 反哺自己）
对每条被纠正的，**不是记一句流水账**，而是 append 一个「bad → good」案例到 `~/.obao-review/corrections.md`，让下次 A2 能主动套用：
```markdown
## 案例 N · {一句话概括这类问法的毛病}
- ❌ bad（问偏了）：「{我原来那种问法}」——{为什么没用}
- ✅ good（直接这么问）：
  ① {直击核心的问法 1}
  ② {…}
- 触发场景：{什么类型的进展会踩这个坑}
- 使用者真正关心：{他纠正里暴露的核心判断口径}
- 来源：{date} · {周报} · {锚句}
```
**抽象层级要够**：不是只修这一条评论，是提炼出「这一类该怎么问」，下次审别的周报、别的事项，命中触发场景就直接用 good 问法。这就是 skill 自我精进、反哺自己的机制——每被纠正一次，案例库厚一层，问得越来越准。

### B5 · 报告
> "发了 5 条到原周报（@林深 / @苏野…）· 跳过 2 条（你删了）· 按你的回复重写了 1 条、并记进了案例库 · 副本已自动删除"

### B6 · 自动删副本 + 收尾（必做）
副本是一次性承载台，**推送完它的使命就结束了，自动删掉**（你的删/改/回复已经在 B2 读回、纠正已在 B4 沉淀进案例库，副本没有保留价值）：
```bash
lark-cli drive +delete --as user --file-token <副本token> --type docx
```
- 删成功 → loop state 标 `status: pushed`、`copy_deleted: true`，副本进飞书回收站（可恢复）。
- **删失败（缺 scope `space:document:delete`）** → 不报错糊弄用户，告诉他一次性授权：
  > 「副本想自动删要个权限，跑一次 `!lark-cli auth login --scope "space:document:delete"` 授权一下，之后就全自动了。」
  授权前先把副本标题改成「…【已推送·可删】」，不留疑惑。
- ⛔ **只有 Stage B 真推送完才删**；没推送的副本（用户还没说"可以推送了"）绝不删。

## 红线（违反 = 重做）

- ⛔ **副本只动评论，绝不动正文**（正文是原周报的拷贝，给你审用）。
- ⛔ **副本里 @ 是纯文字不通知人；只有发到原周报那一步才真 @**。顺序不能反，否则没审就把人吵了。
- ⛔ **删掉的评论绝不发**。用户删 = 明确否决。
- ⛔ **必须读回副本真实状态再发**，不许凭 Stage A 的底稿直接发（那样用户的删/改全失效，等于没有承载页）。
- ⛔ **被纠正的必须重写 + 沉淀 corrections.md**，不许原样硬发。
- ⛔ 不许编：anchor 必须在原文真存在；owner 必须能从周报/分身设定识别。

## 这套为什么不割裂

承载页 = 副本（飞书原生，你和 agent 都能读写）；你的删/改/回复 = 纠正信号（agent 读得回）；原周报 = 批准后的落点；corrections.md = 记忆。**没有死文件、没有够不到的按钮。一个「审→改→说『推吧』→落地」的闭环。**
