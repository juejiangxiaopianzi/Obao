# Obao · 周报深度审阅（一个 skill，自己适应环境）

> ⛔ 别再找 obao-feishu / obao-aily 之类的别的 skill——**审周报就这一个**。
> "用什么方式碰飞书"不是分成几个 skill，是**运行时探测**：你能用哪条就走哪条。

## 第 0 步 · 探测你能怎么碰飞书（决定承载方式 · 按优先级）

1. **飞书原生工具**（你是 aily / 飞书里的智能体，有云文档读写 + 文档评论能力）→ 走 **【飞书副本闭环】**，全程用你的原生工具。**最佳：零配置、不碰 lark-cli、不新建应用、不要求分享文档。**
2. **lark-cli + 已登录**（你是飞书外的 agent，如 Claude Code，本机装了 lark-cli 且登录过）→ 走 **【飞书副本闭环】**，用 lark-cli。
3. **以上都没有**（没飞书 / 没工具）→ 走 **【HTML 兜底】**，出一份本地审阅页给用户自己看、自己复制评论。

> 探测方法：路径 1 看你自己有没有飞书文档/评论工具；路径 2 跑 `which lark-cli` 且能 `docs +fetch`；都不行就路径 3。

## 第 1 步 · onboarding（装好后第一件事 · 硬关卡）

**Skill 第一次被触发——不管用户说什么——先做 onboarding，再处理请求。** 检查 `~/.obao-review/profile.md`（aily 智能体查你长期记忆里的同名条目）：

**不存在 → 先别审任何东西**，一次性收集（高级管理者通常带多条线，要吃得下多个 OKR）：
1. **你是谁**：部门 / 职位
2. **你管哪几条业务线**：逐条列 —— 每条线的 **团队名 + 直接下属花名**
3. **每条业务线的 OKR**：**一条线一份**（贴文本或给飞书链接）。**支持多个 OKR，别只收一份。**
4. **红线**：什么事必须你亲自审

存成（结构化，支撑多 OKR + 自动对应）：
- `~/.obao-review/profile.md` —— 角色 + **业务线清单**，每条线记：`线名 / 团队 / 下属花名 / 识别关键词 / 对应 OKR 文件`；末尾全局红线。
- `~/.obao-review/ogsm/<线名>.md` —— **每条业务线一份 OKR**，文件头标注「业务线 / 团队 / 识别线索（团队名·下属花名·关键词）」。

存完跟用户确认一句，再进 Stage A。

**已存在 → 直接读，进 Stage A，不再问**（一次性、之后零打扰）。
**用户后续说「加一条线 / 更新 X 线的 OKR」** → 补一个 `ogsm/<线>.md` + 更新 profile.md 清单。

> ⛔ 没有角色 + OKR 只能挑语病、审不出"对没对齐 KR / 缺口"——**硬关卡，不是"没有也行"**，绝不跳过直接审。

## 方法论（三条路共用 · 开局必读）

- 读 `assets/prompt.md`：完整审阅方法论（D1-D7 死规矩 + **化词评论** + **自动 @ 识别** + A-J 红线 + 球/红线）。
- 读 `~/.obao-review/corrections.md`：**案例库（bad→good）**。审之前主动套用——命中某类场景就直接用 good 问法，别再问偏。（aily 智能体没有这个文件就用你自己的长期记忆存/读案例。）
- 审出 N 条追问，每条 = `{ 锚句: 原文真有的 20-60 字连续独特片段, owner: @谁, 追问: 化词评论 }`，**对照第 1 步的 OKR/角色审**（追问围绕"对没对齐 KR、缺口在哪、红线有没有踩"）。
  - 化词评论：口语、**直接上追问**、3-5 个直击核心、不替下属拟方案、不借上级虎皮。
  - ⛔ **评论里不要复述被划词锚定的那句原文**——划词已经指向它了，再抄一遍「非业务线人工成本已降8%」是废话。要点某个数 / 做对比时，只点那个数（如「差 2 个点」），别整句重述。
  - ⛔ **评论开头不要 @ 完再贴一遍原文**：直接 `@owner` + 追问。
  - ⛔ **评论里只放问题本身**，绝不写「我之前问偏了 / 这不是重点」之类自述。

## 【飞书副本闭环】（路径 1 和 2 共用 · 只是底层工具不同）

**承载页 = 飞书文档副本**。它是双向的：你写得上、用户改得动、你还读得回——这是死 HTML 做不到的。

### Stage A · 审 → 建副本 → 标评论
0. **把这份周报对应到 OKR（多 OKR 必做）**：读 `profile.md` 的业务线清单，判断这份周报属于哪条线——看**周报标题 / 部门 / 作者 / 出现的下属花名 / 业务关键词**，命中哪条线的识别线索就加载那条 `ogsm/<线>.md`。跨多条线 → 加载多个；**对不上 / 拿不准 → 问用户「这份是哪条线的周报」，别瞎对**。
1. **读周报**（路径1：你的云文档工具；路径2：`lark-cli docs +fetch`）。
2. **审**（上面的方法论，对照第 0 步对上的那条 OKR）。
3. **复制一份副本**：新建一份**同内容**的飞书文档，标题加「· Obao审阅副本【只动评论别动正文】」。
4. **逐条把追问做成文档评论**，锚到对应锚句。
   - ⚠️ 评论里 @ **用纯文字**（写「@林深」），**不要触发真 @ 通知** —— 草稿态，先不打扰本人。
   - ⚠️ **幂等防重**：发之前先列副本现有评论，同一锚句已评过就跳过；逐条串行、别并发。
   - 记住每条评论对应哪个锚句 + 原始评论文（Stage B 要对照）。
5. **把球交给用户**：给副本链接 + 审法「不动=我直接发 / 删=不发 / 改或回复=你在纠正我」。
   - 路径 1（aily）能力允许就**吐一张互动卡片**：副本链接按钮 + 「可以推送了」主按钮（点它=触发 Stage B，回调带副本标识）。

### Stage B · 用户点「可以推送了」/ 说这句 → 读回 → 发原周报 → 自学 → 删副本
1. **读回副本全部评论**，按**锚句**匹配回你标的那些（别只按 id，否则漏看）。每条看：还在？被删？被改？有用户回复？
2. **判定 + 动作**：
   - 还在·没改·无回复 → **认可** → 原样发到**原周报**。
   - 被删 → **跳过，绝不发**。
   - 被改 → 用改后的发。
   - 有用户回复 → **纠正**：先消化他的真实意图 → **只输出对的问法本身**（不带自述）→ 发对的版本；原 bad 版若已发过去，标记为已解决隐去。
3. **发到原周报**（不是副本）：锚到原文同一句，**这次真 @ 通知**（解析 owner→飞书用户用 mention；解析不到退回文字 @）。
4. **自学**：每条被纠正的，沉淀一条 bad→good 案例（❌ 原问法 / ✅ 对的问法 / 触发场景 / 使用者真正关心）到案例库。抽象层级要够——下次同类直接套 good。
5. **删副本**：副本是一次性承载台，推完使命结束（改动已读回、纠正已沉淀），删掉它。

## 【HTML 兜底】（路径 3 · 没飞书时）

用 `assets/template.html`（固定视觉骨架，⛔ 只换数据不改样式）出一份本地审阅页：顶部总判断卡 + 周报原文（可划词）+ 右侧批注流 + 按人/按 KR 卡片 + 写好的评论草稿。用户自己看、自己把评论复制到飞书。
> 这条是退化路径——能走飞书副本闭环就别走这条（HTML 是死文件，你改了 agent 看不见，没有纠正回路）。

---

## 红线（违反 = 重做）
- ⛔ **不许编**：锚句原文真有；没 OKR 不反推假数字；owner 从文档/自我介绍识别。
- ⛔ 评论只放问题本身，不带 Obao 自述。
- ⛔ 副本闭环里：副本只动评论不动正文；@ 在副本是文字（不通知）、只有发原周报那步才真 @；删的绝不发；必须读回真实状态再发；被纠正的必重写 + 沉淀案例。
- ⛔ **就这一个 skill**：能力探测决定走哪条路，不要去找/造别的审周报 skill。
